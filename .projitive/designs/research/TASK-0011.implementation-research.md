# TASK-0011 Implementation Research Brief

## Task
评估并设计多 schema 备份支持

## Summary
评估 PostgreSQL 同时备份多个 schema 的必要性、配置方式和执行边界，在不破坏当前简单 YAML 设计的前提下，给出最小可行扩展方案与验收标准。

---

## Background and Motivation

当前配置仅支持单个 schema 字段，适合大多数情况：
- 备份整个数据库
- 只备份一个 schema
- 按表筛选

但在部分 PostgreSQL 场景下，一个数据库会包含多个业务 schema，用户可能只想备份其中几个，而不是整个库。这时多 schema 支持就有明确价值。

不过这类需求的普遍性低于多目的地，因此应作为次优先级能力推进。

---

## Product Requirements

### Must Have
1. 保持现有全库备份行为不变
2. 保持单 schema 配置继续可用
3. 如需新增多 schema，配置应尽量简单，例如 `schemas: [public, audit]`
4. 与 `tables` 的组合规则必须明确且可预测
5. 文档中清楚说明恢复边界与使用限制

### Should Have
1. 支持 schema 与 schema.table 混合筛选的明确优先级
2. 在路径模板和日志中保留 schema 维度信息

### Non-goals
1. 不做跨数据库聚合备份
2. 不做自动 schema 发现与复杂匹配表达式
3. 不为少数场景引入过重 DSL

---

## Recommended Config Direction

建议采用最小扩展：

```yaml
name: app-subset-backup
source:
  type: postgresql
  connection:
    host: localhost
    port: 5432
    username: postgres
    password: ${DB_PASSWORD}
  database: app
  schemas: [public, audit]
```

规则建议：
- `schema` 与 `schemas` 互斥
- 未设置时表示全库
- 如设置 `tables`，允许使用完整限定名，例如 `audit.logs`

---

## Code Architecture and Impact Areas

### Relevant Files
- `source/core/types.ts` - 为来源配置增加 `schemas?: string[]`
- `source/core/scanner.ts` - 校验 `schema` 与 `schemas` 的互斥关系
- `source/adapters/database/postgresql.ts` - 为 `pg_dump` 增加重复 `-n` 参数支持
- `source/core/executor.ts` - 保持输出 key 与日志的稳定性

### Implementation Notes
1. PostgreSQL 本身支持多次传入 `-n` 参数，技术实现相对直接
2. 需要重点定义与 `tables` 的组合行为，避免歧义
3. 恢复场景可能仍以整份 dump 为主，不一定需要额外接口变化

---

## Acceptance Criteria

1. 单 schema 与全库备份无回归
2. 多 schema 配置可通过校验
3. 生成的 `pg_dump` 参数包含多个 `-n`
4. 带有跨 schema 表名的表筛选行为符合文档说明
5. README、示例配置和测试用例完整更新

---

## Priority Recommendation

相较于多目的地，这项能力更适合作为第二阶段增强：
- 需求真实，但覆盖面较小
- 实现不复杂，但需要清楚定义行为边界
- 建议在多目的地设计稳定后推进

---

## Open Questions

1. 是否还需要支持通配形式的 schema 选择
2. 当 `tables` 与 `schemas` 同时出现时，最终导出集合如何描述得最直观
3. 文件命名中是否需要体现多个 schema 信息

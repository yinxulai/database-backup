# TASK-0010 Implementation Research Brief

## Task
评估并设计多目的地备份支持

## Summary
支持单次数据库导出后，将同一份备份产物分发到多个目的地，以满足异地容灾、双云冗余和本地归档等常见场景，同时保持当前普通 YAML 的简洁配置风格。

---

## Background and Motivation

当前实现一次任务只能写入一个目的地。这在简单场景下足够，但在生产中有几个常见诉求：

1. 同时保存到对象存储和本地磁盘
2. 同时保存到两个对象存储桶或两个云厂商
3. 主备份成功后，希望附带生成归档副本

相比配置两份独立任务分别执行，多目的地支持的核心价值是：
- 只执行一次数据库 dump，避免重复压库
- 同一份备份内容统一校验与命名
- 降低运维配置重复度

---

## Product Requirements

### Must Have
1. 保持现有单目的地写法继续可用
2. 提供最小扩展的多目的地写法，避免引入复杂嵌套结构
3. 单次任务中数据库只导出一次，再复用同一产物上传到多个目的地
4. dry-run 能覆盖所有目的地的可达性与配置有效性检查
5. 日志中能清晰看到每个目的地的上传结果
6. 任一目的地失败时，要有明确且一致的任务状态语义

### Should Have
1. 支持目标类型混用，例如 S3 与 local
2. 每个目的地允许独立 pathPrefix
3. 保留策略可按目的地单独执行

### Non-goals
1. 不做复杂任务编排系统
2. 不引入类 Kubernetes 的包装结构
3. 不在 v1 中支持条件路由、复制链路或异步重试队列

---

## Recommended Config Direction

优先保持单目标配置不变，仅为高级场景增加列表形式：

```yaml
name: prod-backup
source:
  type: postgresql
  connection:
    host: localhost
    port: 5432
    username: postgres
    password: ${DB_PASSWORD}
  database: app

destinations:
  - type: s3
    s3:
      endpoint: https://s3.amazonaws.com
      region: us-east-1
      bucket: prod-primary
      accessKeyId: ${AWS_ACCESS_KEY_ID}
      secretAccessKey: ${AWS_SECRET_ACCESS_KEY}
  - type: local
    local:
      path: /data/archive
```

兼容策略建议：
- 保留 `destination` 作为单目标简写
- 新增 `destinations` 作为数组写法
- 两者互斥，避免歧义

## Code Architecture and Impact Areas

### Relevant Files
- `source/core/types.ts` - 扩展目标配置类型
- `source/core/scanner.ts` - 校验单目标与多目标互斥规则
- `source/cli/run.ts` - 解析多个目标配置
- `source/core/executor.ts` - 实现单次 dump、多路上传 fan-out
- `source/adapters/storage/s3.ts` - 复用现有上传逻辑

### Implementation Notes
1. 执行器需要把 dump 流缓存或 tee 到多个上传流
2. 默认失败语义建议为：任一目标失败则整体任务失败
3. 日志输出中应包含 destination index 或 destination name
4. 如果后续要支持 retention，需基于每个目标分别清理

---

## Acceptance Criteria

1. 单目标配置无回归
2. 多目标配置校验通过
3. 单次执行只触发一次数据库导出
4. 两个及以上目标能收到相同备份内容
5. 任何一个目标失败时，结果和日志符合预期
6. 新增单元测试与执行器回归测试
7. README 与示例配置同步更新

---

## Open Questions

1. 是否需要允许为目标命名，例如 `name: primary-s3`
2. 部分成功时是否保留已上传产物，还是做补偿删除
3. dry-run 是否只做 dump 验证，还是也应验证目标写权限

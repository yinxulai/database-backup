## Code Architecture and Implementation Findings

### 1. DatabaseDriver Interface (source/core/interfaces.ts#L40-L56)
```typescript
export interface DatabaseDriver {
  readonly type: string
  testConnection(): Promise<boolean>
  dump(options: DumpOptions): Promise<Readable>
  close(): Promise<void>
}
```
**Extension needed**: Add `restore()` method for restore functionality.

### 2. PostgreSQLDriver (source/adapters/database/postgresql.ts)
- Uses `pg_dump` for backup via child_process spawn
- Supports gzip compression
- Creates environment with PGPASSWORD
- **pg_restore** is the counterpart for restore - works with custom format (-Fc) dumps
- pg_restore can do point-in-time recovery with `--wal-recovery-target` and `--recovery-target-time`

### 3. StorageDriver Interface (source/core/interfaces.ts#L59-L77)
```typescript
export interface StorageDriver {
  readonly type: string
  upload(data: Readable, key: string): Promise<UploadResult>
  delete(key: string): Promise<void>
  list(prefix?: string): Promise<StorageObject[]>
}
```
**Extension needed**: Add `download()` method to retrieve backup files for restore.

### 4. S3StorageDriver (source/adapters/storage/s3.ts)
- Uses @aws-sdk/client-s3
- Has `list()` but no `download()` method
- Need to add GetObjectCommand for downloading

### 5. BackupExecutor (source/core/executor.ts)
- DefaultBackupExecutor.executeTo() performs:
  1. testConnection()
  2. dump() → computeChecksum() → upload()
- **Need**: restore() method that reverses this flow:
  1. download() from storage
  2. decompress if gzip
  3. restore() to database

### 6. CLI (source/cli/run.ts)
- Uses parseArgs for command parsing
- Supports: run, validate, version, help
- **Need**: add `restore` command

## Implementation Plan

### Phase 1: Type Definitions (source/core/types.ts)

1. Add `RestoreOptions` interface:
   - `backupKey`: S3 key of the backup to restore
   - `targetTime`?: Date for PITR (point-in-time recovery)
   - `tables`?: string[] for partial restore
   - `clean`?: boolean (drop objects before recreating)
   - `create`?: boolean (create database before restoring)

2. Add `RestoreResult` interface:
   - `id`, `taskName`, `status`, `startTime`, `endTime`, `duration`, `error`

3. Add `RestoreStatus` type: `'pending' | 'running' | 'completed' | 'failed'`

4. Add `BackupObject` type for listing available backups

### Phase 2: DatabaseDriver Interface (source/core/interfaces.ts)

1. Add `restore(options: RestoreOptions): Promise<Readable>` to DatabaseDriver
   - Returns a stream that writes to the database (or pipe pg_restore stdin)

### Phase 3: PostgreSQLDriver Implementation (source/adapters/database/postgresql.ts)

1. Add `restore()` method using `pg_restore`:
   - Supports `--clean` (drop objects first)
   - Supports `--create` (create database)
   - Supports `--dbname` for target database
   - Supports `-t` for table filtering
   - Returns Writable stream (pg_restore stdin)

2. pg_restore key arguments:
   - `-h`, `-p`, `-U` for connection
   - `--clean` for clean restore
   - `--create` for database creation
   - `-d` for target database
   - `-f -` outputs to stdout (we use stdin instead)

### Phase 4: StorageDriver Enhancement (source/core/interfaces.ts)

1. Add `download(key: string): Promise<Readable>` to StorageDriver

### Phase 5: S3StorageDriver Implementation (source/adapters/storage/s3.ts)

1. Add `download()` method using GetObjectCommand
2. Returns Readable stream

### Phase 6: BackupExecutor Enhancement (source/core/executor.ts)

1. Add `restore(config, options)` method:
   - Download backup from storage
   - Decompress if needed
   - Pipe to pg_restore
   - Return RestoreResult

### Phase 7: CLI Enhancement (source/cli/run.ts)

1. Add `restore` command:
   - `backup restore --config <file> --backup-key <key>`
   - Options: `--clean`, `--create`, `--tables`

## Risks and Open Questions

1. **pg_restore format**: pg_dump defaults to plain SQL, but pg_restore needs custom format (-Fc). Current implementation uses plain SQL with gzip. Need to decide:
   - Option A: Require custom format for restore (change dump to -Fc)
   - Option B: Support plain SQL restore (psql instead of pg_restore)

   **Decision**: Support both - if backup is plain SQL use `psql`, if custom format use `pg_restore`.

2. **PITR (Point-in-Time Recovery)**: Requires WAL files and is complex. Initial implementation will not support PITR, only full/partial restore.

3. **Data safety**: Restore is destructive. Should add confirmation prompt or `--force` flag in CLI.

4. **Transaction safety**: pg_restore runs in transaction by default with -1, but clean restore may need careful ordering.
